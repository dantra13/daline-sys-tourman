This note restructures section `2.1.1 Competition schedule / Competition schedule update` from the OWG2026 General Messages Interface Document into a readable reference for Sportivo domain modeling. This is the general interface contract for `DT_SCHEDULE` and `DT_SCHEDULE_UPDATE`; sport data dictionaries still define sport-specific status triggers, code values, and any sport-specific differences.

## 2.1.1 Competition Schedule / Competition Schedule Update

`DT_SCHEDULE` is a bulk discipline-level message. Arrival of a new `DT_SCHEDULE` resets all previous schedule information for the discipline, so previous `DT_SCHEDULE_UPDATE` messages should be discarded.

`DT_SCHEDULE_UPDATE` is a partial update message. It contains only schedule data being modified, and it updates only the event unit, phase, or session entities included in the message. No change is implied for omitted units, phases, or sessions.

OVR updates the status of each competition unit through `DT_SCHEDULE_UPDATE`.

When schedule data is generated by the Schedule Management application before the Games, `Competition/ExtendedInfos/ExtendedInfo` may include schedule application metadata:

| Type | Code | Value meaning | Expected OVR flow |
|---|---|---|---|
| `CS` | `VERSION` | Version details from the competition schedule application. | Not expected to be sent to OVR or from OVR. |
| `CS` | `STATUS` | Status details from the competition schedule application. | Not expected to be sent to OVR or from OVR. |

## Early Schedule Stages

The competition schedule starts as a high-level Schedule by Day/Session years before the Games, then becomes progressively more detailed until sessions and competition units are fully defined.

In early stages, the message can describe sessions, events, or phases to provide a high-level view by venue and day. These early messages use `DocumentSubtype="PRE"`.

For `PRE` schedule messages:

- Phase RSCs from the common-code `PHASE` set and event RSCs from the `EVENT` set may appear in `DT_SCHEDULE`, even when their common-code schedule flag would not normally include them.
- The same RSC can appear multiple times in one `DT_SCHEDULE` message.
- To identify a row uniquely during this period, use `Unit/@Code + Unit/@StartDate + Unit/@Venue`.
- `Session/@StartDate` and `Session/@EndDate` may be date-only values when the time is not confirmed yet.
- `Unit/@StartDate` and `Unit/@EndDate` are not included when the unit is scheduled only by session type.
- `Unit/@HideStartDate` is not applicable to `DocumentSubtype="PRE"` messages.

## Detailed Schedule Stage

Once the detailed competition schedule is available, the message contains complete schedule information for all event unit and phase RSCs related to the competition.

At this stage, all `EVENT_UNIT` and `PHASE` codes whose common-code schedule flag is `Y` or `S` are included in schedule messages regardless of status. Units without a concrete status are sent as `UNSCHEDULED` when their schedule flag is `Y` or `S`.

After control of the competition schedule is transferred to OVR, central systems may send a full resynchronisation message with `DocumentSubtype="SYNC"`.

## Head-to-Head Start Lists

`StartList` is included only when the unit type is one of the following and at least one competitor or progression rule is known:

| Unit type | Meaning | Examples from general document |
|---|---|---|
| `HATH` | Individual head-to-head units | Archery, badminton, tennis, snowboard, etc. |
| `HCOUP` | Pairs/couples head-to-head units | Badminton, tennis, etc. |
| `HTEAM` | Team head-to-head units | Basketball, beach volleyball, handball, curling, ice hockey, football, etc. |

`Composition`, meaning the athlete list under a competitor, is only included for `HATH` or `HCOUP`.

## Managing Unknown Start Times

Some disciplines manage units by order rather than by a fixed start time. In those cases, multiple units can share the same `StartDate`, with subsequent units flagged as `HideStartDate="Y"` and ordered by `Unit/@Order`.

Scheduled start and end dates must not be overwritten with actual times. Use `ActualStartDate` and `ActualEndDate` once the unit has actually started or finished.

If display text is needed for a hidden start, use `StartText`. Typical values include free text such as `Not before 17:00`, `SUN 29 - 2nd match on CC`, or `Follows`. When `StartText` uses `Not before hh:mm`, the sent `StartDate` is expected to use the same `hh:mm`.

For team sports, `HideStartDate="Y"` is only used temporarily to remove times; it is not the normal order-management mechanism.

## Trigger and Frequency

| Message or condition | Trigger/Frequency |
|---|---|
| `DT_SCHEDULE` before the Games | Sent as a bulk message when available. Early versions contain high-level sessions and events; later versions contain phase and event-unit detail. |
| Detailed pre-Games schedule | Sent multiple times until a date to be confirmed; after that, OVR sends only update messages. |
| `DT_SCHEDULE` after transfer of control | May be triggered by central systems with `DocumentSubtype="SYNC"` for ODF client resynchronisation. |
| `DT_SCHEDULE_UPDATE` | Triggered whenever schedule data changes for previously sent bulk or update content, including addition of H2H start-list details. |
| H2H start-list details | Sent immediately when officially known, usually as soon as possible after the preceding unit becomes official. |
| Status-change triggers | Defined in each sport data dictionary when a sport has differences. |
| Text-only description changes | A message is not resent just to correct old descriptions; the new description is used in future messages. |

## Message Sort

When the discipline is defined with event-order mode `LOC`, sort units as follows:

1. Day, or filter by day.
2. Session code.
3. Location.
4. Time, regardless of `HideStartDate="Y"`.
5. `Order`.

When the discipline is defined with event-order mode `SESSION` or `DATE`, sort units as follows:

1. Day, or filter by day.
2. Session code.
3. Time, regardless of `HideStartDate="Y"`.
4. `Order`.

`Order` can be the match number for simple display ordering when multiple matches share the same start time.

## XSD and Publication Drift Notes

This general document is `OWG2026-GEN-4.6 APP`, dated 10 December 2025. The football `DT_SCHEDULE` reference currently in the repo was extracted from a Paris 2024 FBL data dictionary. Treat this file as the general interface baseline for OWG2026, but validate integrations against the exact XSD and sport dictionary release used by the target event.

The change log for this general document indicates explicit `DT_SCHEDULE` interface changes after the Paris 2024-era material, including `PreviousValue`, removal of modification indicators, `DocumentSubtype="PRE"`, `HideUnitNum`, `MediaAccess`, phase support in `PreviousUnit/@Unit`, and guide elements. The extracted pages 17-32 do not show a separate `Start/@PreviousValue` row in the message-values table, so verify this field against the target XSD/release before implementation.

The related XSD ZIP available for this PDF material contains `odf2-schema-30112025-DRAFT`, not a schema package explicitly labeled `OWG2026-GEN-4.6 APP`. Keep the PDF version and schema package identity separate in references and validation reports. See `2026-05-26-gen-xsd-package-30112025-draft.md`.

For this draft XSD package, the most relevant `DT_SCHEDULE` drift points are:

- `MediaAccess` appears in the PDF/change control but is absent from `scheduleUnitType`.
- `Start/@PreviousValue` appears in change control but is absent from `scheduleStartType`.
- `Session/@StartDate` and `Session/@EndDate` are `xs:dateTime` in the XSD, while the PDF allows date-only values in early `PRE` schedule messages.
- `Unit/@PhaseType` is required by `scheduleUnitType`, while the PDF describes conditional use and says it should not be sent for non-sport activities when not applicable.
- `Start/Competitor` is optional in `scheduleStartType`, while the PDF message-values table describes it as required.

## Modeling Notes

- Model `DT_SCHEDULE` as a discipline-level replacement snapshot.
- Model `DT_SCHEDULE_UPDATE` as a partial patch over included units, phases, and sessions only.
- The general interface now explicitly allows phase-level schedule rows, not only event-unit rows.
- Early-stage `PRE` schedule rows can represent sessions, events, or phases and can repeat the same RSC. Do not key early-stage schedule rows by `Unit/@Code` alone.
- Keep scheduled start/end times separate from actual start/end times.
- `HideStartDate`, `HideEndDate`, `HideUnitNum`, and `StartText` are presentation contracts that affect display and ordering.
- `MediaAccess` confirms that non-competition schedule units can exist in the general interface and can be open or closed to media.
- Non-competition units use a 34-character RSC-like pattern and free-text `ItemName` / `ItemDescription`; this is the hook for schedule items such as meetings, press conferences, draws, ceremonies, or similar non-match activities when the sport/event implementation supports them.
- For football/team H2H schedules, `StartList` can carry team competitors or progression placeholders. Athlete composition is not normally present because composition is limited to `HATH` and `HCOUP`.
- `PreviousUnit/@Unit` can point to phases as well as event units in OWG2026.
- `Start/@PreviousValue` appears in the OWG2026 document control changes, but not in the extracted message-values table for pages 17-32; treat it as publication drift to verify against schema and integration samples.
- `Guide` elements apply to athlete compositions in applicable para/disability contexts.
